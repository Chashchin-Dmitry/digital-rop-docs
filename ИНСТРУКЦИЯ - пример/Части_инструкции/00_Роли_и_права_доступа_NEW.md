# Роли и права доступа

**Пользователи. Выбор прав.png** - Управление ролями пользователей в системе

Система разграничения доступа основана на четырех основных ролях с различными уровнями функциональности. Каждая роль определяет объем доступных данных и возможности управления системой.

## Роли пользователей

### Супер-администратор

{INTERFACE} Максимальный уровень доступа с правом управления всеми аспектами системы. Видит данные всех офисов и менеджеров, управляет техническими настройками, API интеграциями и правами пользователей. Единственная роль с доступом к настройке слов-паразитов.

### Администратор

{INTERFACE} Расширенные права управления с возможностью настройки чек-листов, создания аналитических таблиц и управления тестированием. Доступ к техническим параметрам и веб-хукам. Может утверждать тесты и редактировать их содержание.

### Руководитель офиса

{INTERFACE} Управление командой конкретного офиса с доступом к данным только своих подчиненных. Основные функции включают просмотр статистики офиса, аналитики менеджеров и обязательное утверждение тестов перед отправкой сотрудникам.

### Менеджер

{INTERFACE} Базовый уровень доступа к собственным данным: просмотр своих звонков с оценками, транскрипций разговоров, прохождение утвержденных тестов и отслеживание персональной статистики. Не имеет доступа к данным коллег.

## Доступ к данным

### Ограничения по офисам

**Коммуникации. Фильтр по офису.png**

{TECHNICAL} Система автоматически фильтрует данные в зависимости от роли пользователя. Руководители офисов видят только информацию по своему подразделению, менеджеры получают доступ исключительно к собственным записям, администраторы работают с полным объемом данных компании.

### Территориальное разграничение

**Менеджеры. Фильтр по офисам.png**

{INTERFACE} В интерфейсе фильтрации руководители офисов видят только свое подразделение, администраторы получают список всех офисов компании. Менеджеры не имеют доступа к фильтрам по офисам, поскольку видят только собственную статистику.

## Управление тестированием

### Workflow утверждения тестов

**Тесты. 1 - редактирование теста 2 - одобрить тест для отправки в аккаунт менеджеру.png**

{TECHNICAL} Автоматически сгенерированные тесты требуют обязательного утверждения руководителем перед отправкой менеджеру. Руководитель может отредактировать формулировки вопросов и ответов, после чего одобрить тест для прохождения. Только утвержденные тесты становятся видимыми сотрудникам.

### Прохождение тестов менеджерами

**Тесты. пример теста.png**

{INTERFACE} Менеджеры видят только утвержденные руководителем тесты в статусе "Не пройден". После выбора ответа и отправки результата тест переходит в статус "Пройден" с отображением правильного ответа и пояснений.

## Административные функции

{TECHNICAL} Разделы "Настройки" и "Дополнительные настройки" доступны исключительно администраторам и супер-администратору. Включают управление API ключами, настройку чек-листов, создание аналитических таблиц, управление пользователями и конфигурацию интеграций с внешними системами.

## Интеграция с LDAP

{TECHNICAL} Система использует корпоративные учетные данные Setl Group для аутентификации. Автоматическое создание пользователей при первом входе, синхронизация с организационной структурой и автоматическое отключение доступа при изменении статуса сотрудника в CRM.

## Управление доступом

**Пользователи. общий вид.png**

{INTERFACE} Назначение ролей осуществляется в разделе [Настройки - Пользователи](#настройки-пользователи). Супер-администратор может назначить любую роль, обычные администраторы управляют ролями менеджеров и руководителей. Изменения вступают в силу при следующем входе пользователя в систему.
