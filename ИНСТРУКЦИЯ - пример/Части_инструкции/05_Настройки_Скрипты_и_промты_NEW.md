# 5. Настройки - Скрипты и промты

Раздел для настройки чек-листов и критериев анализа звонков. Основа для создания [тестов](#аналитика-тесты) и оценки качества работы [менеджеров](#аналитика-менеджеры).

## Общий вид

**Скрипты и промты. Общий вид. Настройка "Шаблоны скрипта".png**

{INTERFACE} Страница содержит три основные вкладки: "Шаблоны Скрипта" для настройки чек-листов анализа звонков, "Настройка Конверсии" для определения целевых действий и "Настройка Тегов" для создания тематических меток классификации звонков.

### Список шаблонов скриптов

{INTERFACE} В таблице отображается готовый шаблон "Чек-лист (тест 001)" с настроенным вебхук URL для отправки результатов анализа. Для каждого шаблона доступны кнопки "Изменить" и "Удалить" для управления настройками.

### Функции управления

- **"Добавить шаблон"** - создание нового чек-листа
- **Редактирование** - изменение существующих шаблонов  
- **Удаление** - удаление неиспользуемых шаблонов

## Пример настроенного шаблона

**Скрипты и промты. Пример настроенного шаблона скрипта.png**

Детальный просмотр настроенного чек-листа с этапами оценки.

{INTERFACE} Шаблон содержит три основных этапа: "МРК выяснил все критерии поиска клиента", "МРК выяснил финансовую ситуацию клиента" и "МРК выяснил ситуацию клиента (цель покупки И что важно)". Каждый этап имеет детальное описание критериев выполнения и настройки оценки.

### Структура этапов

{TECHNICAL} Каждый этап содержит:
- **Название этапа** - краткое описание проверяемого действия
- **Критерии выполнения** - детальное описание когда этап считается выполненным
- **Условия игнорирования** - случаи когда этап не учитывается в оценке
- **Строгость оценки** - процентный порог для получения +1 или -1 балла

## Создание нового шаблона

**Скрипты и промты. Добавить шаблон скрипта.png**

{INTERFACE} Форма создания шаблона содержит поля для названия шаблона, описания назначения и URL для отправки результатов анализа в CRM. После заполнения основной информации можно переходить к добавлению этапов проверки.

## Пошаговое добавление этапа

### Шаг 1. Название этапа

**Скрипты и промты. Добавить этап. 1 - название этапа.png**

{INTERFACE} Указывается краткое название проверяемого действия. Название должно четко отражать суть проверки и быть понятным для менеджеров. Кнопки "Назад" и "Вперед" позволяют управлять процессом создания.

### Шаг 2. Критерии выполнения

**Скрипты и промты. Добавить этап. 2 - когда считаем выполненным (описание этапа и критерии выполнения этапа, за что +1 балл начисляется).png**

{INTERFACE} Детальное описание того, когда этап считается выполненным. В большом текстовом поле указываются конкретные критерии, ключевые слова и действия, за которые начисляется +1 балл при анализе звонка.

### Шаг 3. Условия игнорирования  

**Скрипты и промты. Добавить этап. 3 - когда этап игнорируется из оценки.png**

{INTERFACE} Настройка случаев, когда этап не учитывается в общей оценке звонка. Описываются ситуации, при которых невыполнение этапа не влияет на итоговый балл менеджера.

### Шаг 4. Строгость оценки

**Скрипты и промты. Добавить этап. 4 - от 0 до 100 насколько строго оцениваем этап.png**

{TECHNICAL} Настройка порога выполнения этапа в процентах. Если ИИ определяет выполнение этапа выше указанного процента - начисляется +1 балл, ниже - начисляется -1 балл. Рекомендуемые значения: 50-70%.

## Настройка тегов

**Скрипты и промты. Настройка тегов.png**

{INTERFACE} Вкладка для создания тематических меток звонков. Показан пример настроенного тега "Что такое холодный звонок?" с техническим названием тега и промтом для определения классификации.

### Структура тега

{TECHNICAL} Каждый тег включает:
- **Название** - описательное наименование тега
- **#Тег конверсии** - техническое название для системы  
- **Промт** - инструкция для ИИ по определению тега

## Принцип работы системы

{TECHNICAL} После создания шаблона скрипта ИИ анализирует каждый звонок против настроенных этапов. Система проверяет выполнение критериев, учитывает условия игнорирования и применяет настройки строгости для присвоения баллов. Результаты отправляются в CRM через [настроенные webhook](#api-документация).
