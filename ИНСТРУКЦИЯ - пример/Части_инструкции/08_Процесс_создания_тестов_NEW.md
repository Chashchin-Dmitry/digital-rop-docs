# Процесс создания и управления тестами

Описание полного цикла создания, утверждения и прохождения тестов в системе "Цифровой РОП".

## Общий принцип работы

**Тесты. Общий вид.png**

{INTERFACE} Система автоматически создает персонализированные тесты для каждого менеджера на основе выявленных ошибок в звонках. Тесты отображаются в разделе "Аналитика → Тесты" с указанием статуса, менеджера и возможностями управления.

## Автоматическое создание тестов

{TECHNICAL} Алгоритм работает следующим образом: система анализирует звонок менеджера по настроенному чек-листу, выявляет невыполненные этапы, выбирает ОДНУ наиболее критичную ошибку и формирует тест на основе этой ошибки. Даже если в звонке было несколько ошибок, создается только один тест.

### Процесс генерации

{TECHNICAL} Для создания теста система использует:
- **Вопрос** - формируется на основе названия невыполненного этапа чек-листа
- **Правильный ответ** - берется из описания "когда считаем этап выполненным"  
- **Неправильные ответы** - генерируются ИИ (всегда 3 варианта)
- **Объяснение** - создается на основе критериев выполнения этапа

## Workflow утверждения тестов

### Статус "Не одобрен"

{INTERFACE} Сразу после создания тест получает статус "Не одобрен" и НЕ отображается у менеджера. Тест виден только руководителю офиса в разделе "Аналитика → Тесты" для проверки и утверждения.

### Редактирование теста

**Тесты. Пример редактирование теста.png**

{INTERFACE} Руководитель может отредактировать тест перед утверждением. В режиме редактирования доступно изменение формулировки вопроса и всех 4-х вариантов ответов. Важное ограничение: количество вариантов всегда строго 4 (нельзя добавить или удалить).

### Утверждение теста

**Тесты. 1 - редактирование теста 2 - одобрить тест для отправки в аккаунт менеджеру.png**

{INTERFACE} После проверки руководитель нажимает кнопку одобрения (зеленая галочка). Тест меняет статус с "Не одобрен" на "Не пройден" и становится доступен менеджеру для прохождения.

## Прохождение тестов менеджером

**Тесты. пример теста.png**

{INTERFACE} Менеджер видит в своем интерфейсе только утвержденные тесты со статусом "Не пройден". При открытии теста отображается вопрос с 4 вариантами ответов. После выбора ответа и нажатия "Отправить результат" тест переходит в статус "Пройден".

### Обратная связь

{INTERFACE} После прохождения теста менеджер получает результат с указанием правильного ответа и подробным объяснением. Это помогает понять ошибку и избежать её повторения в будущих звонках.

## Управление тестами руководителем

### Фильтрация по статусам

**Тесты. Фильтр по статусу теста.png**

{INTERFACE} Руководитель может фильтровать тесты по статусам:
- **"Не одобрен"** - ожидающие проверки и утверждения
- **"Не пройден"** - утвержденные, отправленные менеджеру  
- **"Пройден"** - выполненные сотрудниками

### Мониторинг команды

{INTERFACE} Фильтры по менеджерам и датам позволяют отслеживать прогресс обучения каждого сотрудника и оперативно реагировать на проблемные области в работе команды.

## Связь тестов с практикой

{TECHNICAL} Каждый тест напрямую связан с конкретным звонком менеджера, где была допущена ошибка. Это обеспечивает персонализированное обучение на основе реальных ситуаций из практики сотрудника.

### Цикличность процесса

{INTERFACE} Процесс непрерывный: после прохождения теста менеджер возвращается к работе, система продолжает анализировать новые звонки и при выявлении ошибок создает новые тесты для дальнейшего обучения.

## Ключевые особенности системы

1. **Автоматизация** - тесты создаются без участия человека
2. **Персонализация** - каждый тест основан на реальной ошибке менеджера
3. **Контроль качества** - обязательное утверждение руководителем
4. **Гибкость** - возможность редактирования перед утверждением
5. **Практичность** - связь с конкретными ситуациями из работы
